# 版本回退方法总结

## 一、为什么需要版本回退

在 CI/CD 流程中，即使经过了充分的测试，生产环境仍可能出现不可预期的问题。常见的回退场景包括：

1. **代码缺陷**：测试未覆盖的边界情况导致的线上故障
2. **性能问题**：新版本导致系统响应变慢或资源消耗过高
3. **功能回滚**：某些功能需要暂时下线
4. **配置错误**：错误的配置导致服务异常

## 二、版本回退策略

### 2.1 Blue-Green 部署策略

Blue-Green 部署维护两套相同的环境：
- **Blue 环境**：当前生产环境
- **Green 环境**：新版本部署环境

```
回退流程：
Green 环境发现严重问题 → 切换流量到 Blue 环境 → Green 环境回退修复
```

**优点**：回退速度快，几乎无停机时间
**缺点**：需要双倍资源成本

### 2.2 Canary 部署策略

先向小部分用户推送新版本，逐步扩大范围：
```
10% 用户 → 30% 用户 → 50% 用户 → 100% 用户
   ↓           ↓           ↓         ↓
  正常       正常       发现问题   (已回退)
```

**优点**：问题影响范围可控
**缺点**：回退需要逐步进行

### 2.3 Rolling Update 策略

逐个替换实例，而不是同时替换所有实例：
```
旧版本容器: [v1] [v1] [v1] [v1]
              ↓
新版本容器:   [v2] [v1] [v1] [v1]
              ↓
新版本容器:   [v2] [v2] [v1] [v1]
              ↓
新版本容器:   [v2] [v2] [v2] [v1]
              ↓
新版本容器:   [v2] [v2] [v2] [v2]
```

**优点**：无需额外资源
**缺点**：回退时需要重新部署

## 三、具体回退方法

### 3.1 Git 代码回退

#### 方法 1：git revert（推荐）

创建一个新的提交来撤销之前的提交，保留完整的提交历史：

```bash
# 查看提交历史
git log --oneline -10

# 撤销指定提交（会创建新的撤销提交）
git revert abc1234

# 推送到远程
git push origin main
```

#### 方法 2：git reset

回退到指定版本，不保留后续提交：

```bash
# 软重置（保留修改在暂存区）
git reset --soft abc1234

# 混合重置（保留修改在工作区）
git reset --mixed abc1234

# 硬重置（丢弃所有修改，慎用！）
git reset --hard abc1234

# 强制推送到远程
git push --force origin main
```

**警告**：硬重置会丢失未提交的修改，请确保已暂存或备份。

### 3.2 Docker 镜像回退

#### 从本地镜像回退

```bash
# 1. 查看镜像历史
docker images | grep simple-web

# 2. 停止当前容器
docker compose down

# 3. 标记旧镜像为最新
docker tag <image-id> simple-web:latest

# 4. 重新启动
docker compose up -d

# 5. 验证
curl http://localhost/health
```

#### 从镜像仓库回退

```bash
# 1. 列出可用的版本标签
docker images | grep simple-web
# 或使用镜像仓库命令
docker search <registry>/simple-web

# 2. 拉取指定版本
docker pull <registry>/simple-web:v1.0.0

# 3. 重新部署
docker compose down
docker tag <registry>/simple-web:v1.0.0 simple-web:latest
docker compose up -d
```

### 3.3 GitHub Actions 自动回退

#### 方法 1：重新运行成功的 Workflow

```bash
# 1. 打开 GitHub 仓库
# 2. 点击 Actions 标签
# 3. 找到最近一次成功的 Workflow Run
# 4. 点击 "Re-run all jobs"
```

#### 方法 2：回退代码后触发新流水线

```bash
# 1. 本地回退代码
git reset --hard <good-commit-hash>

# 2. 推送到远程
git push --force origin main

# 3. GitHub Actions 会自动触发新流水线
```

### 3.4 阿里云 ECS 直接回退

SSH 登录到服务器执行：

```bash
#!/bin/bash
# 回退脚本

set -e

echo "=== 开始回退流程 ==="

# 1. 进入部署目录
cd /opt/simple-web

# 2. 停止当前服务
echo "[1/5] 停止当前服务..."
docker compose down || true

# 3. 清理旧容器和镜像（可选）
echo "[2/5] 清理旧资源..."
docker system prune -f

# 4. 重新拉取或加载镜像
echo "[3/5] 加载镜像..."
docker load -i simple-web.tar.gz

# 5. 启动服务
echo "[4/5] 启动服务..."
docker compose up -d

# 6. 健康检查
echo "[5/5] 健康检查..."
sleep 5
HTTP_CODE=$(curl -s -o /dev/null -w "%{http_code}" http://localhost:80)

if [ "$HTTP_CODE" = "200" ]; then
    echo "✅ 回退成功！HTTP: $HTTP_CODE"
else
    echo "❌ 回退失败，HTTP: $HTTP_CODE"
    docker compose logs --tail=50
    exit 1
fi

# 显示当前状态
echo ""
echo "=== 当前运行状态 ==="
docker compose ps
```

## 四、回退最佳实践

### 4.1 回退前

1. **确认问题**：明确是否真的是代码问题，排除配置、网络等因素
2. **评估影响**：评估回退对业务的影响
3. **通知相关方**：告知相关团队成员即将进行回退
4. **准备回退方案**：准备好回退脚本和验证方法

### 4.2 回退中

1. **记录操作**：记录每一步操作，便于问题排查
2. **保持通信**：及时向相关方同步回退进度
3. **监控指标**：监控关键指标，确保回退过程正常

### 4.3 回退后

1. **验证服务**：确认服务正常运行
2. **通知完成**：通知相关方回退已完成
3. **分析根因**：分析导致问题的根本原因
4. **修复验证**：修复后进行充分测试再重新部署

## 五、快速回退检查清单

```
□ 确认问题范围和影响
□ 通知团队成员
□ 选择合适的回退方法
□ 执行回退操作
□ 验证服务正常
□ 确认功能正常
□ 记录回退过程
□ 分析问题根因
□ 制定修复计划
□ 安排后续测试和部署
```
